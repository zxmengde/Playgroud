# 长任务质量标准

本文件规定 Codex 处理长任务、跨会话任务和中断恢复任务时的质量标准。核心原则是：任务越长，越需要状态保存、局部验证、恢复入口和独立检查。

## 适用范围

以下任务应按长任务处理：

- 预计超过 30 分钟的调研、系统改造、正式报告、PPT、代码实现或多文件整理。
- 涉及多个来源、多个工具或多个可交付文件的任务。
- 会被速率限制、网络、权限、上下文压缩或用户离开打断的任务。
- 用户明确要求“继续工作”“做到底”“不要停在中间”的任务。

## 关键判断

外部评测显示，agent 在长任务中的主要困难通常不是单步知识不足，而是多步执行中的状态丢失、早期路径错误、验证不足和恢复能力不足。对个人工作系统而言，长任务能力不应只用“运行了多久”衡量，而应看是否持续保存状态、不断验证中间产物，并能在中断后恢复到正确位置。

## 运行要求

长任务开始时，Codex 应建立最小上下文地图：

- 用户真实目标和成功标准。
- 当前工作区、关键规则、已知偏好和未知偏好。
- 关键来源、输入文件、输出文件和不应修改的范围。
- 风险边界和需要确认的操作。
- 可运行的验证方式和可能失败点。

执行过程中，应在自然检查点记录状态。自然检查点包括：完成一组来源阅读、完成一个文件修改、运行一次校验、发现阻塞、提交或同步前后、上下文即将变长或用户离开前。

记录位置优先使用 `docs/tasks/active.md`、知识条目、复盘记录或专门任务状态文件。不要只把状态保存在对话中。

## 验收维度

长任务结束前，应至少从以下维度检查：

- 需求满足：产物是否回应了用户真实目标，而不是只完成字面步骤。
- 回归避免：修改是否破坏已有规则、技能、模板、脚本或用户偏好。
- 步骤完整：关键步骤是否有明显遗漏，是否停在计划或中间文件。
- 证据质量：来源是否可靠，事实、推断和不确定性是否区分。
- 鲁棒性：是否考虑了网络失败、速率限制、缺失文件、隐性数据缺损和工具失败。
- 可恢复性：中断后能否从本地记录继续，而不是依赖用户重复说明。
- 可审计性：文件路径、提交、验证命令和剩余风险是否清楚。

## 中断恢复

中断恢复时，Codex 应先读取本地任务记录、知识条目、Git 状态和最近提交，再继续执行。不得要求用户重新描述已经写入本地记录的信息。

恢复顺序为：

1. 检查当前时间、工作目录和 Git 状态。
2. 读取 `AGENTS.md`、`docs/tasks/active.md`、相关知识条目和最近提交。
3. 判断哪些工作已完成、哪些已验证、哪些仍未同步。
4. 先补齐未提交、未验证或未同步的低风险工作。
5. 再继续新的调研、修改和记录。

## 自我评分

长任务完成后，Codex 应在必要时给出简短自我评分，使用以下状态之一：

- 通过：产物、验证、记录和同步均完成。
- 部分通过：核心产物完成，但仍有明确剩余风险或外部阻塞。
- 未通过：未形成可用产物，或验证显示目标未满足。

自我评分不替代用户判断，只用于防止 Codex 在未完成时提前结束。
