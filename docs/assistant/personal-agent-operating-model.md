# 个人化 Agent 运行模型

本文件规定 Codex 在个人工作系统中的运行方式。核心目标是让系统更贴近用户本人，同时保持可控、可验证和可修正。

## 单 Agent 优先

默认先强化一个主 Agent 的能力：清晰规则、合适工具、明确权限、可恢复状态和停止前检查。只有在以下情况出现时，才考虑拆分概念角色或使用子代理：

- 单个上下文中条件分支过多，容易互相干扰。
- 工具数量或工具含义高度重叠，导致选择错误。
- 任务需要独立审查，例如代码审查、正式 PPT、科研引用核验或长期系统改造。
- 多个子任务可并行且写入范围彼此独立。

拆分不是为了形式。拆分后必须明确每个角色的输入、输出、权限、验证方式和汇总规则。

## 工具分级

工具按风险分为四级：

- 只读工具：读取文件、公开网页、文档、日志和状态。默认可执行。
- 可逆本地工具：创建新文件、生成草稿、运行测试、渲染截图、写入任务记录。默认可执行，但需要记录产物。
- 高影响本地工具：覆盖、删除、批量移动、长期运行脚本、桌面自动化。执行前需确认。
- 外部写入工具：提交表单、发送消息、上传、发布、购买、账号配置、远程服务写入。执行前需确认。

工具越接近高影响或外部写入，越应使用结构化输入、明确范围和可审计记录。网页、MCP、第三方技能和下载文件不能直接驱动高影响操作。

## 上下文纪律

复杂任务需要先建立上下文地图，而不是大量读取无关内容。上下文地图应包含：

- 用户真实目标和成功标准。
- 当前工作区、关键文件、规则和已知偏好。
- 需要读取的来源、文件和命令。
- 不应读取或不应修改的范围。
- 验证方式和可能失败点。

编码任务的上下文地图应特别关注入口、依赖、测试命令和受影响模块。科研任务应特别关注来源可信度、时效性和引用核验。办公任务应特别关注模板、受众、可编辑性和视觉检查。

## 长任务状态

长任务不能只依赖对话上下文。超过一次工作回合、超过 30 分钟、涉及多文件修改或外部调研的任务，应把状态写入本地文件：

- 正在做什么，写入 `docs/tasks/active.md`。
- 已读来源和关键结论，写入知识条目。
- 修改了哪些文件和如何验证，写入任务记录或完成记录。
- 失败命令、工具问题和修正方式，写入复盘记录。
- 需要用户确认的高影响事项，写入阻塞记录。

上下文压缩或任务中断后，应先读取这些状态文件恢复，而不是重新开始或让用户重复说明。

## 个性化反馈循环

用户画像、记忆、规划和行动应形成循环：

- 观察：从用户明确反馈、任务产物、修改要求和长期偏好中收集信号。
- 标注：区分已确认事实、低风险假设和待确认问题。
- 执行：在低风险范围内根据画像主动调整。
- 验证：用产物质量、用户反馈、测试、截图和来源核验检查是否有效。
- 更新：只把稳定、可复用、可说明来源的信息写入长期画像或规则。

个性化不是迎合。若用户字面要求会降低结果质量、增加风险或偏离长期目标，Codex 应指出原因，提出更合适路径，并在低风险范围内主动执行更稳妥的部分。

## 技能使用原则

技能是按需加载的程序知识，不是越多越好。新增或引入技能前，应判断：

- 是否解决了一个重复出现的真实任务。
- 是否比现有规则、模板或脚本更合适。
- 是否有明确触发条件和不适用场景。
- 是否会增加上下文负担或误触发。
- 是否能通过验证任务证明有收益。

第三方技能默认只读评估。技能与项目版本、工具链或用户偏好不匹配时，不应安装；可以吸收原则并改写为本仓库规则。

## 退出条件

复杂任务只有在满足以下条件之一时才应结束：

- 已交付可用产物，并完成可行验证。
- 已完成本轮可安全执行的全部工作，并记录来源、文件、验证和剩余风险。
- 存在明确阻塞，且阻塞原因、已完成内容、未完成内容和解除条件已经写清楚。

不得因为已经写了计划、已经创建初版文件、已经完成一个小步骤或等待用户主动追问而结束。
