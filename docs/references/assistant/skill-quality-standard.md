# 技能质量标准

本文件规定仓库内 skills 的设计、审查和校验标准。目标是让 skills 成为可复用能力，而不是更长的提示词集合。

## 基本原则

每个 skill 应聚焦一个职责。名称和描述应清楚说明它做什么、何时使用、输入是什么、产出是什么。描述决定触发可靠性，不应模糊或覆盖过宽。

`SKILL.md` 应保持简洁。复杂背景、模板、示例、API 细节和长流程应放入 `references/`、`assets/` 或 `scripts/`，并由 `SKILL.md` 按需指向。

## 自由度设置

技能应根据任务风险设置自由度：

- 高自由度：适用于研究、写作、代码审查、方案设计等需要结合上下文判断的任务。
- 中自由度：适用于有固定结构但需要按输入调整的任务，例如报告、表格分析、文档整理。
- 低自由度：适用于易出错或风险较高的任务，例如迁移、发布、批量改动、外部写入和权限相关操作。

低自由度任务应优先使用脚本、检查项和明确命令，不应只靠自然语言约束。

## 必备内容

每个 skill 至少应包含：

- `name`：小写字母、数字和连字符，长度不超过 64。
- `description`：说明做什么和何时使用，长度不超过 1024。
- 触发场景和不适用场景。
- 读取哪些文件或上下文。
- 产出什么。
- 如何验证。
- 何时需要任务级授权或预授权。

## 禁止模式

不得创建以下类型的 skill：

- 名称或描述过于宽泛，导致几乎所有任务都会触发。
- 把用户必须知道的内部机制暴露为使用前提。
- 要求保存密钥、令牌、账号密码或敏感身份信息。
- 绕过权限、忽略安全边界或自动执行外部写入。
- 直接复制来源不明的第三方 skill、泄露源码或安装脚本。
- 只有建议，没有可执行流程、产物或校验方式。

## 校验要求

新增或修改 skill 后，应运行 `scripts/validate-skills.ps1`。该脚本检查 frontmatter、名称、描述长度、文件长度，并在本机存在 `quick_validate.py` 时调用官方校验脚本。

技能通过格式校验不代表已经适合长期使用。真实任务后还应观察触发是否准确、是否过度加载上下文、是否减少返工、是否提升产物质量。

## 经验验收

外部评测表明，skills 的收益依赖任务匹配、上下文兼容、版本时效和验证方式。新增或修改 skill 后，除格式校验外，还应至少在一个代表性任务中检查：

- 是否在正确场景触发，而不是过度触发。
- 是否减少重复说明和返工。
- 是否没有引入与项目版本、工具链或用户偏好冲突的指导。
- 是否有清楚的产物和验证结果。

若一个 skill 只能增加提示长度，不能稳定改善真实任务，应简化、合并或移除。

## 成本纪律

技能不是越多越好。新增技能应减少重复说明、减少临时查找或提升验证质量。若一个技能只是把常识写得更长，或导致过度触发、过度读取和长篇输出，应合并到现有工作流或删除。

技能正文应优先写触发、输入、执行、产物、验证和权限边界。背景知识、长示例和可变细节应放入按需读取的引用文件，或保留为脚本和模板。
