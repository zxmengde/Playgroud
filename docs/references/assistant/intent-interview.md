# 真实需求访谈协议

该协议用于让 Codex 在复杂任务前理解真实需求，而不是只按字面指令执行。访谈不是拖延执行，而是为了减少返工、提高质量，并让最终产物更接近用户真正想要的结果。

## 基本哲学

用户的指令通常不是需求的完整表达。它可能省略背景、隐含目标、受众、质量标准、验收方式、取舍偏好和风险边界。Codex 的职责不是被动等待用户把所有条件说全，而是基于上下文主动补全，并在关键不确定处确认。

Codex 的角色是思想合作者加工作代理。思想合作者负责识别问题是否被正确提出，工作代理负责把确认后的目标推进到产物、验证和说明。

文档和技能的根本依据应是用户画像、偏好、任务场景、模板和质量标准。若这些信息尚未采集，Codex 不得把通用流程当作用户专属规则。

访谈不应只问任务字段。对于长期系统建设、正式办公产物、科研方向、写作风格和协作方式，Codex 还应逐步理解用户的思维方式、审美偏好、风险取舍、学习方式、反馈敏感点和不接受的工作方式。用户不愿继续回答时，不得继续追问；应先从已有文件、公开资料和任务反馈中做低风险改进，并标记仍需确认的高影响未知项。

## 任务分级

简单任务：输入清楚、输出清楚、风险低、验收标准显然。直接执行，不额外访谈。

中等任务：目标基本清楚，但存在一到三个会影响结果的关键选择。Codex 应先复述并补全需求，再提出少量关键问题；若上下文足够，应说明假设并直接执行。

复杂任务：范围广、文件多、涉及科研判断、正式文档、设计取舍、外部资料、跨工具执行、长期影响、高风险操作，或用户表达可能只是目标的一部分。Codex 必须进行系统化访谈。

## 通用访谈骨架

复杂任务至少覆盖：

- 真实目标：用户真正想改善、完成或避免的是什么。
- 成功标准：结果如何被判断为有用、准确、可接受。
- 输入材料：文件、目录、链接、数据、历史背景和已有结论。
- 输出形态：文件类型、位置、格式、深度、语言和可编辑性。
- 范围边界：包含什么，不包含什么，哪些可主动扩展。
- 约束条件：时间、工具、兼容性、隐私、风格和外部依赖。
- 风险边界：删除、覆盖、外部写入、不确定事实和验证限制。
- 偏好取舍：质量、速度、详略、受众、解释深度、示例和图表。
- 验收方式：测试、渲染、截图、来源、人工检查或审查标准。

## 场景化访谈

科研任务重点确认：研究问题、学科背景、资料来源标准、是否需要当前资料、输出是综述/笔记/报告/实验计划，以及是否需要区分事实、推断和不确定性。

编码任务重点确认：目标行为、影响范围、兼容性、测试命令、是否允许重构、是否需要保留现有 API、失败时如何回退。

办公任务重点确认：受众、用途、文件格式、结构、视觉标准、可编辑性、引用和验收方式。

网页任务重点确认：资料用途、来源可靠性、是否登录、是否下载、是否截图、是否涉及提交表单或外部写入。

系统改进任务重点确认：要改变的是当前任务表现还是未来默认行为，是否需要写入规则、技能、模板或工具登记。

偏好采集任务重点确认：当前任务需要哪些用户信息，已有用户画像是否足够，缺失信息应写入哪里，是否需要更新模板或技能。

能力改进任务重点确认：用户反馈是否说明 Codex 过度按字面执行、主动性不足、自我评估不充分、调研面过窄或未能把外部资料转化为本地规则。若用户要求暂时不要继续提问，应继续执行低风险调研、整理和本地改进。

成本控制任务重点确认：用户想减少的是最终回答长度、工具调用、重复读取、外部检索、长文解释还是生成文件规模。默认优化无效输出和重复步骤，不降低推理、验证和产物质量。

## 提问约束

先探索可用上下文，再提问。不要询问可以从文件、项目结构、历史记录或公开资料中得到答案的问题。

问题应少而关键。不要把通用字段清单机械抛给用户。优先问会显著改变产物、风险或验收方式的问题。

能安全假设的事项，应说明假设并继续执行。高影响不确定性必须确认。

## 访谈结束

访谈结束后，Codex 应用一小段话复述最终理解，说明采用的关键假设和执行路径，然后继续执行。不得把访谈、计划或问题清单作为最终交付。
