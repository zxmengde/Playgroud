# 小型综合个人工作代理能力路线

本文件把“用户应得到怎样的同伴”转化为能力路线。它与 `docs/capabilities/index.md` 配合使用：index 记录当前成熟度，本文件记录下一步建设顺序。

## 设计原则

能力路线按任务价值和风险排序，不按工具数量排序。一个能力只有能减少真实协调成本、增加可验证产物或改善恢复能力，才进入建设清单。

每项能力都应有触发场景、输入、输出、权限边界、验证方式和复盘入口。缺少这些要素时，应先保持流程文档或脚本候选状态。

## 近期优先级

第一优先级是降低恢复成本：保持 `AGENTS.md`、核心协议、任务状态和知识索引一致，新增文件必须接入校验。

第二优先级是补齐真实样例：Zotero、视频、GitHub、浏览器、Office 和编码任务各保留至少一个可复核验收记录。

第三优先级是工具稳定性：修复 GitHub 代理，建立 MCP 试用规则，避免外部工具失效时影响本地工作链。

第四优先级是精简：把兼容入口、归档文件、样例输出和重复画像列为候选项，确认后再合并或删除。

## 能力提高方式

Codex 应通过真实任务和校验提高能力，而不是靠增加长规则。有效改进应留下至少一种产物：脚本、skill、模板、验收记录、知识条目或复盘记录。

当一个流程连续多次使用且输入输出稳定时，应优先写成脚本；当流程需要文字规则和示例时，应写成 skill；当流程定期重复且失败可发现时，再转为 automation 候选。
