# 能力差距审查

本文件记录当前个人工作系统距离高能力个人助手的差距。它用于指导后续改进，不作为宣传性说明。

## 结论

当前系统已经具备基本规则、权限边界、知识沉淀、科研、办公、网页、编码、Zotero 文献和视频资料入口。但这些能力仍处于可用骨架阶段，不应被视为成熟形态。

主要差距不在于模型是否足够强，而在于本地系统是否给模型提供了稳定输入、可靠工具、可执行流程、可验证标准和持续改进机制。若缺少这些支撑，模型能力会被本地环境、资料缺失、网络失败、任务状态丢失和输出冗余消耗掉。

## 当前技能成熟度

技能成熟度按四级判断：

- 草稿：有触发说明，但缺少真实任务验证。
- 可用：能指导任务，并有明确产物和验证方式。
- 稳定：经过多个真实任务验证，能减少重复说明和返工。
- 自动化候选：流程稳定、输入输出明确、失败可发现，可考虑周期性运行。

当前多数技能处于“可用”或“草稿到可用之间”。`execution-governor`、`style-governor`、`research-workflow`、`coding-workflow` 和 `office-workflow` 已有稳定方向，但仍缺少真实任务样例集。`literature-zotero-workflow` 和 `video-source-workflow` 是新增能力，必须在真实 Zotero 和视频任务中验证后才能提高成熟度。

## 关键差距

### 真实需求判断

现有规则已经写入反迎合、伪需求判断和主动扩展，但仍缺少失败样例库。后续需要记录用户实际不满意的回答、正确替代路径和验收结果，使规则从抽象原则变成可复用判断模式。

### 工具接入

Zotero、Bilibili、浏览器、GitHub、Office 和本地文件目前主要依赖通用工具和流程说明。要接近高能力个人助手，需要把高频工具变成可运行脚本或稳定 MCP/CLI 路径，例如 Zotero 只读导出、字幕抓取、引用核验、PPT 渲染检查和 Git 网络诊断。

### 环境稳定性

GitHub 网络问题会直接削弱同步、提交和远程协作能力。当前诊断表明 Git 代理配置本身存在，但 Clash/Mihomo 代理端口不可达，且普通 Codex 进程无法重启系统服务。该问题必须通过管理员权限重启服务、重新应用 Clash 核心或调整代理监听来解决。

### 评估机制

技能格式校验只能证明文件结构正确，不能证明任务效果好。需要新增技能审计脚本、技能成熟度记录、真实任务样例和失败复盘。对于复杂技能，应优先在真实任务中验证，而不是继续增加自然语言规则。

### 成本控制

已有成本控制文档，但还缺少可量化检查。后续应观察最终回复长度、重复读取、工具输出噪声和是否把应写入文件的内容放在对话中。成本控制应减少无效输出，不能减少核验和验证。

## 改进路径

第一步，建立技能审计：每个技能至少说明触发、读取上下文、产物、验证和权限边界。新增 `scripts/audit-skills.ps1` 用于发现缺项。

第二步，建立真实任务样例：为科研文献、Zotero、Bilibili、PPT、代码修改和网页资料各保留一到两个可复用样例，记录输入、执行路径、产物和验证。

第三步，工具化高频任务：把 Zotero 只读导出、字幕获取、引用核验、PPT 渲染检查、Git 网络诊断做成脚本或稳定命令，而不是每次临时推理。

第四步，自动化维护：当技能审计、知识索引检查、Git 网络检查和复盘摘要稳定后，再考虑 Codex 自动化。官方 Codex 文档建议先把重复流程做成技能，稳定后再自动化；这与本仓库方向一致。

第五步，保持反迎合执行：每次复杂任务结束前检查是否只是完成了字面步骤，是否检查了前提和伪需求，是否给出了可验证产物。

## 本轮可执行事项

本轮已完成或正在完成：新增技能审计脚本、记录当前能力差距、更新 Git 网络诊断文档、把用户反馈写入复盘记录，并运行系统校验。
