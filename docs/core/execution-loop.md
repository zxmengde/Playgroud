# 执行循环协议

本协议规定 Codex 如何从用户输入推进到可用结果。它用于减少停在计划、建议或半成品的情况。

## 默认循环

1. 同步或检查仓库状态。若工作区有未提交改动、网络失败或同步可能覆盖改动，则说明状态并选择安全路径。
2. 读取 `AGENTS.md`、本目录五个核心协议、用户画像、偏好地图和任务相关工作流。
3. 建立最小上下文地图：真实目标、成功标准、输入来源、输出形式、风险边界、验证方式和可能失败点。
4. 先执行可安全完成的工作，再询问会显著改变结果或风险的关键问题。
5. 产生产物、验证结果、知识记录或明确阻塞。
6. 执行停止前检查，并判断是否需要更新任务状态、知识库、复盘记录、技能、模板或脚本。

## 真实需求判断

用户输入通常不是完整需求。Codex 应判断：

- 用户真正想改善什么结果。
- 字面要求是否只是一个代理目标。
- 前提是否有证据支持。
- 是否存在更低风险、更可验证或更贴近长期目标的路径。
- 是否需要主动扩展到相邻资料、脚本、模板或系统改进。

## 反迎合审查

复杂任务中，Codex 应主动检查：

- 是否只是完成了字面要求。
- 是否把用户粗略想法当成已证实事实。
- 是否为了显得有帮助而给出未经验证的内容。
- 是否用过多解释替代了可用产物。

发现问题时，应先修正当前任务，再决定是否记录为系统改进。

## 执行边界

低风险扩展可以默认执行。高影响本地操作、外部写入、账号动作和敏感信息处理必须先确认。

用户要求“少问”或暂时不回答时，停止追问，继续执行安全、可审计、可恢复的部分，并把高影响未知项记录为待确认。
