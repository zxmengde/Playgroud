# 用户使用教程

本教程说明如何使用这个受控的个人智能工作系统。核心用法很简单：用户只需要表达目标，Codex 负责同步仓库、读取上下文、理解需求、执行任务、验证结果，并在发现未来行为需要改进时主动提出确认。

## 一、日常使用方式

本机使用时，通常不需要先手动运行 `git pull`。只要在 Codex 中说明要处理本仓库或相关任务，Codex 应主动进入 `D:\Code\Playgroud`，同步仓库，检查状态，然后开始工作。

云端使用时，选择私有仓库：

```text
https://github.com/zxmengde/Playgroud
```

云端 Codex 应主动确认当前分支和远程 `main` 的关系，然后读取仓库规则。用户不需要记住内部目录。

## 二、怎样提任务

可以直接用自然语言提出目标。例如：

```text
帮我把 D:\Code\MEMS 的下周科研任务整理清楚，并写入系统。
```

```text
帮我分析这个项目，找出下一步应该做什么，能验证就验证。
```

```text
把这份材料整理成可编辑的 Word 大纲，并检查结构。
```

```text
总结这个网页，把长期有用的信息保存下来。
```

如果任务很简单，Codex 应直接执行。如果任务较复杂，Codex 会先进行自适应访谈，确认目标、产物、范围、限制、质量标准和验收方式。访谈的目的是避免只按字面指令执行，尽量理解真实需求。

## 三、Codex 会自动做什么

Codex 应默认承担以下工作：

- 同步本地或云端仓库状态。
- 读取项目规则和相关上下文。
- 判断任务属于科研、编码、办公、网页、知识沉淀或系统改进。
- 判断任务复杂度。
- 对复杂任务进行系统化澄清。
- 执行可完成的工作。
- 运行可用校验。
- 记录有长期价值的信息。
- 在确认后修改系统规则、技能或模板。
- 完成校验、提交和推送。

用户不需要记住内部技能名或文件名。内部机制由 Codex 自行使用。

## 四、什么时候会询问用户

Codex 只应在问题会影响结果或风险时询问。

通常会询问的情况包括：目标不清楚、输出形式影响较大、质量标准无法推断、需要外部账号操作、可能删除或覆盖文件、需要发布或发送内容、可能保存敏感信息。

如果可以从文件、上下文或公开资料中得到答案，Codex 应自行查找，不应把问题转交给用户。

## 五、系统如何持续改进

当 Codex 发现自己没有充分理解需求、执行不够彻底、验证不足、语气不合适或工具使用失败时，应先修正当前任务。若该问题可能影响后续工作，Codex 应主动说明建议更新哪类规则，并询问是否确认。

用户可以只说：

```text
这次不符合我的预期。
```

或者：

```text
以后遇到这种情况你应该自己先确认。
```

Codex 应自行判断问题类型，提出修正建议。用户确认后，Codex 自动完成文件修改、校验、提交和推送。

## 六、权限边界

低风险工作由 Codex 主动完成，例如读取文件、检索公开资料、生成草稿、创建新文件、运行测试、整理知识、检查文档结构。

以下操作仍需确认：删除、覆盖、大规模移动、外部账号写入、提交表单、发送消息、购买、发布、安装长期运行服务、保存可能敏感的信息。

以下操作不得执行：保存或复述密钥、令牌、账号密码；绕过权限限制；未经确认修改重要资料；把敏感资料写入公开位置。

## 七、如何判断系统正在正常工作

正常表现包括：

Codex 会先读取相关文件，而不是直接凭空回答。

简单任务会直接执行，不增加额外交互。

复杂任务会先问少量关键问题，以确认真实需求。

任务结束时会给出产物、验证结果或明确阻塞。

如果发现需要改进未来行为，Codex 会主动提出确认，而不是要求用户记住内部文件名。

## 八、同步与提交

用户一般不需要手动同步。Codex 修改本仓库后，应运行校验、提交并推送。若推送失败，Codex 应说明原因，并在可行时使用已授权的 GitHub 接口同步。

如果用户自己在终端修改了仓库，可以使用常规 Git 命令：

```powershell
cd D:\Code\Playgroud
git pull
.\scripts\validate-system.ps1
git status
git add .
git commit -m "Update controlled work system"
git push
```

这只是手动路径，不是用户日常必须记住的步骤。
