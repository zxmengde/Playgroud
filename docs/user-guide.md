# 用户使用教程

这个系统的目标是降低用户负担。你不需要记住内部文件名、技能名或流程名。通常只需要直接说目标，Codex 应自行同步仓库、读取上下文、理解真实需求、必要时澄清、执行、验证，并在需要改进未来行为时主动请求确认。

`D:\Code\Playgroud` 是可信工作区，默认采用高自主模式。Codex 应优先把任务推进到可用产物、验证、记录、提交和同步，而不是因本仓库内可回退操作反复请求许可。

## 如何使用

本机使用时，你可以直接在 Codex 中说：

```text
帮我整理 D:\Code\MEMS 的下周科研任务。
```

```text
帮我把这份材料做成可编辑的 Word 大纲。
```

```text
帮我分析这个项目下一步应该做什么，能验证就验证。
```

Codex 应主动进入 `D:\Code\Playgroud`，同步仓库，检查状态，并读取系统规则。云端使用时，选择私有仓库：

```text
https://github.com/zxmengde/Playgroud
```

云端 Codex 应主动确认当前分支和远程 `main` 的关系，然后读取仓库规则。

## Codex 应如何理解需求

简单任务应直接执行，不增加额外交互。

中等任务应先复述并补全需求，只问会影响结果的一到三个关键问题。

复杂任务应像访谈一样确认真实目标、成功标准、输入、输出、范围、约束、风险、偏好和验收方式。访谈结束后，Codex 应继续执行，而不是停在问题或计划上。

如果你的指令可能限制更好的结果，Codex 应直接说明原因，提出更合适路径，并在低风险范围内主动扩展执行。

如果 Codex 缺少与你个人相关的偏好信息，例如 PPT 模板、Word 格式、科研受众、引用方式或代码习惯，它应主动采集，而不是假装已经知道。

## Codex 应如何交付

复杂任务的交付应同时包含可用产物和必要说明。产物可以是文件、补丁、知识条目、报告、截图或验证结果。说明应解释依据、取舍、验证方式和剩余风险。

Codex 不应只给建议，不应把计划当作最终结果，也不应要求你记住内部机制。

## 系统如何改进

如果 Codex 发现自己没有充分理解需求、执行不彻底、验证不足、语气不合适或工具使用失败，应先修正当前任务。若该问题会影响未来任务，Codex 应主动说明建议改变的默认行为，并询问你是否确认。

你可以只说：

```text
这次不符合我的预期。
```

Codex 应自行判断问题类型，修正当前结果，并在需要时提出系统更新建议。你确认后，Codex 应自动完成文件修改、校验、提交和推送。

## 权限边界

Codex 可以主动执行读取文件、检索公开资料、生成草稿、创建新文件、运行测试、整理知识、检查文档结构等工作。

在 `D:\Code\Playgroud` 内，可由 Git、备份或明确回退步骤恢复的删除、覆盖、大规模移动、提交、推送和分支整理默认已授权。

仓库外不可逆删除或覆盖、外部账号写入、提交表单、发送消息、购买、发布、安装长期运行服务、保存可能敏感的信息，可以通过任务级授权或预授权放开。授权需要说明对象、允许动作、排除项、预算或影响范围、有效期、验证方式和回退方式。授权范围不清时，Codex 应先完成准备工作并停在执行前。

Codex 不得保存或复述密钥、令牌、账号密码，不得绕过权限限制，不得在对象、预算、影响范围或回退方式不清时执行不可恢复外部影响操作。

## 手动路径

用户一般不需要手动同步或提交。如果你自己在终端修改了仓库，可以使用：

```powershell
cd D:\Code\Playgroud
git pull
.\scripts\validate-system.ps1
git status
git add .
git commit -m "Update controlled work system"
git push
```

这只是手动路径，不是日常使用前提。
