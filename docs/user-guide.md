# 用户使用教程

本教程说明如何使用这个“受控的个人智能工作系统”。它的目标不是让 Codex 只记住几条偏好，而是把科研、办公、编码、网页访问、知识沉淀、权限控制和复盘改进组织成一个可持续维护的工作系统。

## 一、这个系统是什么

本仓库 `D:\Code\Playgroud` 是控制仓库。它不是普通笔记目录，而是 Codex 的本地事实来源。Codex 在这里工作时，应先读取 `AGENTS.md`，再根据任务类型读取 `docs/assistant/`、`docs/workflows/` 和 `skills/` 下的规则。

可以把它理解为三层结构。

第一层是行为规则。`AGENTS.md` 和 `docs/assistant/` 规定 Codex 的语言风格、执行深度、权限边界、验证要求和复盘机制。

第二层是工作流程。`docs/workflows/` 规定科研、编码、办公、网页和知识沉淀任务应如何处理。

第三层是可复用资产。`docs/knowledge/` 保存长期知识，`templates/` 保存模板，`scripts/` 保存校验和辅助脚本，`skills/` 保存云端可读的技能定义。

## 二、日常如何开始

本机使用时，先进入仓库：

```powershell
cd D:\Code\Playgroud
git pull
```

然后在 Codex 中提出任务。任务可以直接描述目标，不需要每次重复说明风格和边界。例如：

```text
请按本仓库的受控个人智能工作系统，整理 D:\Code\MEMS 的下一周科研任务。
```

```text
请读取 D:\Code\MEMS\研究总纲.md，把其中的 24 周安排拆成可执行任务，并写入本仓库任务记录。
```

```text
请对 D:\Code\MEMS 做一次项目地图更新，说明入口、模块、依赖、测试命令和风险。
```

```text
请把这篇 PDF 整理成科研知识条目，保留来源、关键词、事实、推断和不确定性。
```

云端使用时，选择或克隆私有仓库：

```text
https://github.com/zxmengde/Playgroud
```

云端 Codex 进入仓库后也会读取 `AGENTS.md`。如果云端无法读取你本机的 `.codex` 技能目录，它仍可读取仓库内 `skills/` 下的技能定义副本。

## 三、应该怎样向 Codex 提任务

最有效的提法是“目标 + 输入 + 期望产物 + 是否需要沉淀”。

例如：

```text
目标：把 MEMS 项目变成下周可执行计划。
输入：D:\Code\MEMS\研究总纲.md 和 D:\Code\MEMS\README.md。
产物：写入 docs/tasks/active.md，并把长期背景写入 docs/knowledge/items/。
要求：按本仓库规则执行，完成后运行校验。
```

如果只是一般任务，也可以更短：

```text
请按本仓库规则处理这个代码问题，能测试就测试，不能测试说明原因。
```

```text
请按本仓库规则总结这个网页，并把有长期价值的信息写入知识库。
```

```text
请按本仓库规则制作一个 Word 大纲，输出到 output/doc/，并检查文件结构。
```

不需要反复说明“用简体中文”“不要停在建议”“注意验证”。这些已经写入规则和技能。如果 Codex 没做到，应直接指出，系统会通过 `harness-log.md` 记录并修正。

## 四、任务会被如何路由

`assistant-router` 是任务路由入口。它会把任务分到不同流程：

科研、文献、PDF、论文和技术调研进入 `research-workflow`。

代码库理解、修改、调试、测试和审查进入 `coding-workflow`。

Word、PPT、Excel、PDF、Markdown 和报告进入 `office-workflow`。

网页访问、截图、资料提取和下载前检查进入 `web-workflow`。

长期偏好、项目背景、术语、模板、资料摘要进入 `knowledge-capture`。

当你指出 Codex 偷懒、理解偏差、语气不合适、验证不足或重复询问时，进入 `harness-capture`。

所有任务都同时受 `style-governor` 和 `execution-governor` 约束。前者控制语言和语气，后者要求任务推进到产物、验证或明确阻塞。

## 五、知识库如何使用

长期有价值的信息写入 `docs/knowledge/items/`。索引在 `docs/knowledge/index.md`。

知识条目使用统一结构：

```yaml
title:
type:
source:
tags: []
status:
paths: []
links: []
next_actions: []
```

常见类型包括：

- `research`：科研主题、文献综述、实验背景。
- `project`：代码项目、目录结构、运行方法。
- `web-source`：网页资料和在线来源。
- `preference`：长期偏好。
- `workflow`：可复用流程。

如果你希望某条信息以后不用重复说明，可以直接说：

```text
请把这条规则沉淀到知识库，并在必要时更新对应技能。
```

或者：

```text
这是长期偏好，请写入本仓库规则，不要只保存在当前对话。
```

## 六、权限边界如何理解

系统把操作分成三类。

可直接执行：读取文件、检索公开资料、生成草稿、创建新文件、运行测试、渲染文档、截图、整理知识条目。

需确认执行：删除、覆盖、大规模移动、外部账号写入、提交表单、发送消息、购买、发布、安装长期运行服务、保存可能敏感的信息。

禁止执行：保存或复述密钥、令牌、账号密码；绕过权限限制；未经确认修改重要资料；把敏感资料写入公开位置。

因此，Codex 应该主动完成低风险工作，但在外部写入和不可逆操作前停下来确认。

## 七、如何让系统持续变好

如果 Codex 表现不符合预期，不要只让它“下次注意”。应要求它修改系统：

```text
这次你停在建议，没有继续执行。请写入 harness-log，并修正 execution-governor。
```

```text
这个语气不符合我的偏好。请更新 preferences.md 和 style-governor。
```

```text
这个任务我已经重复解释过，请把它沉淀为知识条目或模板。
```

复盘记录位于：

```text
docs/assistant/harness-log.md
```

这个文件是系统改进的记录。每次反馈都应对应一个修正动作，例如更新规则、拆分技能、增加模板、增加脚本或补充工具登记。

## 八、本机和云端如何同步

每次修改本仓库后，建议运行：

```powershell
.\scripts\validate-system.ps1
git status
git add .
git commit -m "Update controlled work system"
git push
```

如果先在云端 Codex 修改了仓库，本机使用前先运行：

```powershell
cd D:\Code\Playgroud
git pull
```

如果出现冲突，不要强行覆盖。让 Codex 读取冲突文件，保留两边有效内容后再提交。

## 九、常见任务示例

科研任务：

```text
请按本仓库规则检索玻璃基 TGV MEMS 压力传感器可靠性相关资料，输出一个知识条目，保留来源和不确定性。
```

代码任务：

```text
请按本仓库规则分析 D:\Code\MEMS 的代码结构，更新项目地图，并给出可以直接运行的检查命令。
```

办公任务：

```text
请按本仓库规则把这份研究内容整理成 Word 大纲，输出到 output/doc/，并检查结构。
```

网页任务：

```text
请按本仓库规则访问这个网页，提取关键信息，截图，并将长期有价值的信息写入知识库。
```

风格修正：

```text
这次回答仍然像泛泛建议，没有持续推进。请更新 harness-log 和 execution-governor。
```

知识沉淀：

```text
这是我的长期偏好：以后科研总结必须区分事实、推断和不确定性。请沉淀到本仓库。
```

## 十、如何判断 Codex 是否正确使用了系统

可以检查四点。

第一，是否读取了相关文件，而不是直接凭空回答。

第二，是否产出了文件、补丁、知识条目、截图、报告或验证结果。

第三，是否运行了可用检查，或说明不能检查的原因。

第四，是否在你指出问题后更新了 `harness-log.md` 和相关规则。

如果没有做到，可以直接要求它修正系统，而不是只修正当前回答。
