# 编码工作流程

## 触发场景

适用于代码库理解、功能修改、错误排查、测试、重构、代码审查和技术文档生成。

## 执行要求

开始前读取 `docs/profile/user-model.md` 和 `docs/profile/preference-map.md`。若测试深度、重构边界、提交习惯或项目特定约定未知且会影响任务，先使用 `preference-intake`。若用户暂时不想回答，采用 `preference-map.md` 中的编码保守默认值。

先读取项目结构、依赖文件、测试配置和相关源码。优先使用项目已有风格和工具。改动范围应贴近任务目标，不做无关重构。

复杂代码任务开始前，应生成轻量项目地图：入口文件、关键模块、依赖配置、测试命令、当前分支状态、可能受影响文件和已有约定。项目地图服务于选择读取范围，不能替代真实源码阅读。

需求理解、方案设计、文件编辑和验证应分离。简单任务可以直接编辑；中等和复杂任务应先说明理解与影响范围，再进入修改。编辑后使用测试、构建、静态检查或最小复现进行验证。

若项目存在 `AGENTS.md`、`README`、贡献说明、代码规范、类型配置或测试脚本，应优先读取。若缺少规范，采用项目已有风格，不引入无关抽象。

修改后运行相关测试、构建或静态检查。无法运行时说明原因和剩余风险。

## 产物

产物包括补丁、项目地图、检查命令、测试结果、失败日志摘要和后续阻塞说明。
